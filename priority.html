严重等级评估

基于并行执行策略

从"先做什么后做什么"转变为"哪条链路晚启动 1 天，Go-Live 就晚 1 天？"

一、评估思路

8 条链路 Day 1 全部并行启动，优先级不等于执行顺序。我们关注的核心问题是：

哪条链路晚启动 1 天，Go-Live 就晚 1 天？

为此，我们从四个维度综合评估每条链路的资源投入力度和启动时序，确保外部等待尽早触发、高难度链路预留充足缓冲时间、长周期链路不被低估、强依赖链路优先突破。

二、评估维度（4 个）

#维度定义
1紧急程度
Urgency		外部等待时间越长、越不可压缩，越必须尽早触发——运营商、审批委员会、VP 协调、quota 审批
2推进难度
Difficulty		根因越不清楚，诊断可能需要的迭代轮次越多，应越早启动、预留越多 buffer
3时间跨度
Duration		是否是 Go-Live 的硬性前置条件？整体日历时间越长，越不能低估
4依赖强度
Dependency		此 Blocker 解决后能连锁解锁多少下游工作？依赖越强、牵连越广，越需要先突破
严重等级由紧急程度、推进难度、时间跨度、依赖强度四个维度综合决定

四、伪依赖 vs 真依赖 — 关键识别

表面依赖关系为什么是伪依赖Day-1 可执行替代方案
#1 Interconnect → #5 Pipeline
"带宽修好才能测 Pipeline"		 Pipeline 诊断不需要全量带宽，只有 2PB 全量迁移才需要 用 Cloud Storage 现有数据做 profiling；子集数据即可定位 5a/5c
#5 Pipeline → #6 Region
"Pipeline 修好才能扩资源"		 资源诊断完全独立于 Pipeline，只有 scale-up 负载测试才依赖 当天就能跑完 quota check + 机型测试 + autoscaler 审计；立即提交 quota 申请
#5 Pipeline → BigQuery
"数据写入后才能优化查询"		 BQ 查询效率与 Pipeline 无关，查询优化只需要表结构和样本数据 用现有数据或子集跑最复杂 SQL；INFORMATION_SCHEMA 分析无需新数据
#2 Security Controls → #3 Review
"安全配置修完才能送审"		 审批不必等全部修复再提交，先提交、边做边补 Day 1 提交现有材料（不等完美）；每次修复后更新材料即可
#4 DBA → #8 Adoption
"数据验证完才能推采用"		 采用度 baseline 收集不依赖 DBA，只有 "数据可信" 信号依赖 DBA Day 1 就部署 usage analytics；培训完成率、旧系统流量也可同步收集
#1~#7 技术修复 → #8 验证
"系统能用了才能看有没有人用"		 使用数据现在就能采集；但区分"不想用"还是"不能用"需要等技术修复后 登录数、功能使用率、API 调用量——即使系统部分可用，也能观察使用模式

六、严重等级总结

P0-长线 — #3 Security Review

日历时间最长、Go-Live 硬门槛、不可压缩
原则：Day 1 提交；每次技术修复后更新材料

P0-长线 — #1 Interconnect

2PB 物理传输 ~4 周不可压缩；#4/#5/#6/#7 全量验证全部依赖此链
原则：Day 1 网络诊断 + 当天联系运营商；与 #3 构成 Go-Live 双重硬约束

P0-触发 — #6 Region Resources

诊断当天完成但 quota 审批 2-4 周。12w 跨度（全量验证等 #1）
原则：Day 1 诊断 + 当天提交 quota request（不可压缩等待，越早越好）

P0-短线 — #5 Pipeline Latency

技术不确定性最高，诊断结果驱动后续所有技术决策
原则：子集数据 Day 1 启动 profiling，不等 Interconnect 修复

P1-快赢 — #7 BigQuery Cost

不阻断 Go-Live 但有立竿见影的成本改善价值。12w 跨度（全量验证等 #1）
原则：用数据子集测最复杂 SQL + 查 INFORMATION_SCHEMA → Quick Win

P1-后台 — #8 User Adoption

Day 1 启动后台数据收集，Week 4+ 启动主动验证。12w 跨度
原则：先零成本后台采集数据；技术修复见效后再投入人力做访谈和培训

P1-协调 — #4 DBA Engagement

技术准备 Day 1 启动，人员协调需要 VP 介入。12w 跨度（全量验证等 #1）
原则：CE+Partner 先做技术校验；同步触发 VP 协调 DBA 到位

P1-短线 — #2 Security Controls

Day 1 扫描完成，结果直接喂给 #3 Security Review
原则：当天产出 gap 清单 → IAM/Firewall 快速修复 → Org Policy exemption 申请
