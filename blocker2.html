基本信息
架构图位置GCP Shared VPC → Centralized Security / Data Isolation Zone
架构图标注⚠️ Centralized Security
上下游关系横切所有组件——Org Policy / VPC-SC / IAM / Firewall Rules 影响整个 GCP 项目的资源部署和访问控制
ImpactOrg policy deny 约束可能阻止资源部署；VPC-SC 过严可能阻断数据流通；IAM 权限不足阻止团队操作
假设归属H1: 技术与基建（安全防护的技术配置层面）
怀疑方向组织级安全策略为保护全局安全而设置了严格控制，但这些控制可能过度限制了此项目的资源部署

Discovery — Technical Actions
Action 1Org Policy 全量扫描 — gcloud resource-manager org-policies list --project [PROJECT] + 逐条检查是否有 deny 约束影响所需资源← 2a
Action 2部署测试 — 在目标项目中尝试部署一个最小化的资源集合（GKE node + Cloud Storage bucket + BigQuery dataset），捕获所有 policy violation← 2a
Action 3VPC-SC 审计日志分析 — 查看 audit_log 中的 VPC-SC violation 记录；确认 perimeter 配置和 access level← 2b
Action 4IAM Policy 审查 — 列出项目级和组织级 IAM binding；对比团队成员角色与所需最小权限集← 2c
Action 5Firewall Rules 审查 — 检查 VPC firewall rules 是否阻断了必要的内部通信（如 GKE pod 到 Cloud Storage、Dataflow 到 BigQuery）← 2b
Action 6SCC 合规报告 — 运行 Security Command Center 扫描，生成当前安全态势报告，识别 misconfiguration← 2a,2b,2c
Discovery — Process Actions
Action 7确认 Org Policy 管理 Owner — 谁管组织级 policy？是中央安全团队还是 cloud platform team？← 2a
Action 8确认 Project-level Exemption 机制 — 是否有流程可以为特定项目申请 org policy 豁免或 scoped override？← 2a
Action 9评估 Landing Zone 现状 — 客户是否已有 Landing Zone 标准？如果没有，是否需要用 Cloud Foundation Toolkit 建一个？← 2a,2b,2c
Discovery 产出物
产出物Security Controls Audit Report — 包含：Org Policy 阻断清单、VPC-SC violation 记录、IAM 权限 gap 分析、Firewall 规则审查结果、推荐修复列表（哪些可以 project-level override、哪些需要组织级变更）
执行计划
Phase 0 (Week 1-2)完成 Action 1-9；产出 Audit Report；分类：哪些可立即修复（IAM/Firewall）vs 需要审批（Org Policy override）
Phase 1 (Week 2-3)立即修复：IAM role 补充、Firewall rule 调整；申请 project-level org policy exemption（并行推进）
Phase 2 (Week 4-7)Org policy exemption 获批后实施；用 Cloud Foundation Toolkit 建 Landing Zone 基线（可复用到后续项目）
Phase 3 (Week 8-12)安全配置进入 steady-state；Landing Zone 基线文档化；安全团队签字确认
RACI
R — ResponsibleOutcome CE（审计 + GCP 安全最佳实践指导）+ Customer Cloud Admin（执行配置变更）
A — AccountableHead of Cloud Delivery（安全配置变更审批）
C — Consulted中央安全团队（org policy owner）、Network Engineering（VPC/Firewall）、Google Specialist CE - Security
I — InformedVP of Operations、Compliance 团队
风险与成功标准
成功标准所有必需资源可正常部署（无 policy violation）；VPC-SC 配置允许合法数据流通；IAM 权限满足最小权限原则且不阻碍操作
风险中央安全团队拒绝 project-level exemption；org policy 变更影响范围超出本项目
风险缓解提前准备详细的安全影响评估（为什么需要 exemption、scope 限定在本项目内）；争取安全团队参与 Landing Zone 共建，让他们成为解决方案的一部分而非审批者
Blocker 依赖与 #3（Security Review）关联——安全控制修复后生成的配置是 Security Review 的审批材料之一；与 #6（Region Resources）关联——org policy 可能限制可用 region
⚠️ 伪依赖提示：#3 Security Review 不必等 #2 全部完成——Day 1 扫描出的 gap 清单可立即作为审批输入
